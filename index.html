Azure HR Policy RAG System — Live for 6,000 Employees

Served as the sole architect on an end-to-end Azure RAG system for an HR Policy chatbot now live across the entire organisation of 6,000 employees. I led a 2-engineer team (one frontend, one backend) while personally owning every architecture and infrastructure decision from day one through to production.

The system uses an Angular SPA hosted on Azure Static Web Apps with global CDN delivery, a FastAPI backend on Azure Container Apps with built-in autoscaling and no Kubernetes overhead, and hybrid retrieval via Azure AI Search combining HNSW vector search, BM25 keyword search, and a built-in semantic cross-encoder reranker — all in a single managed API call. GPT-4o runs on Provisioned Throughput Units to guarantee capacity at 200–300 concurrent users with no throttling risk. Azure Document Intelligence handles PDF extraction, correctly processing multi-column layouts, comparison tables as structured Markdown, and embedded flowcharts via GPT-4o Vision descriptions — because HR documents almost always contain tables that basic OCR garbles. All secrets live in Key Vault accessed via Managed Identity, all backend services run on private endpoints inside the Container Apps VNet with no public exposure, and employees authenticate via Microsoft Entra ID with MFA enforced by org policy.

In production the system delivers under 3 seconds end-to-end response time, approximately 40% of queries served from in-memory semantic cache in under 150ms with no LLM call, and a 99.9% uptime SLA. Every answer includes citations with document name, page number, and section title.

Key decisions I made that shaped the architecture: replaced AKS with Container Apps to eliminate Kubernetes operational overhead at this scale, replaced Redis with in-memory caching using cachetools (perfectly sufficient at 200–300 concurrent users, zero extra infrastructure), replaced Cosmos DB with MySQL Flexible Server to reuse the existing chat history schema, replaced Service Bus with Event Grid for direct blob-upload triggers (simpler at 200 PDFs), and dropped HyDE query expansion in favour of PDF filename-based metadata pre-filtering — reducing cost and latency without any measurable retrieval quality loss.

Inference Cost Optimisation — 30% Cost Reduction

LLM inference was running at $15,000 per month with every query hitting GPT-4o regardless of complexity. I co-designed a complexity-based model router that classifies incoming queries at the API gateway level and directs straightforward questions to Gemma and Mistral 8B models while reserving GPT-4o for tasks that genuinely require it. The router uses a lightweight classifier trained on query length, entity density, and reasoning requirement signals. The result was a 30% reduction in monthly inference spend — from $15,000 to $10,500 — with no measurable degradation in answer quality on our golden evaluation set.

LLM Fine-tuning — Hallucination Rate from 45% to 30%

Production evaluation identified a 45% hallucination rate in model responses — answers that were confident but not grounded in the retrieved context. Rather than replacing the base model, I fine-tuned Llama 3 using LoRA/PEFT with Direct Preference Optimisation (DPO) on a dataset of 500+ human-labeled preference pairs where annotators selected preferred responses (well-grounded, cited) over rejected ones (confident but unsupported). The parameter-efficient fine-tuning preserved the base model's general capabilities while significantly improving its tendency to stay within the retrieved context. Post-deployment evaluation showed the hallucination rate dropped from 45% to 30% across production workloads.

EvalOps Pipeline — Zero Undetected Regressions

Before this pipeline existed, model updates were deployed with informal checks and silent regressions were reaching production. I built a full EvalOps pipeline that runs as a mandatory CI gate before any model update can be deployed. It includes groundedness rubrics that verify answers are supported by retrieved context, golden datasets with expected outputs for regression detection, automated RAGAS scoring across faithfulness, answer relevancy, and context precision, and OpenTelemetry retrieval tracing that surfaces chunk-level signals for debugging. Updates that degrade scores below configured thresholds are automatically blocked with a non-zero exit code. Since rollout there have been zero undetected regressions deployed to production.

Retrieval Quality — RAGAS Faithfulness from 0.62 to 0.74

The RAG system was producing answers with a RAGAS faithfulness score of 0.62 on a 500-question evaluation set — meaning the model was frequently drifting from retrieved context and adding unsupported claims. I systematically tuned the hybrid retrieval layer combining BM25 keyword search, pgvector dense retrieval, and cross-encoder reranking, and introduced query rewriting to better align the user's natural language intent with the indexed content. The query rewriter expands ambiguous queries and reformulates them in language closer to how the source documents are written. After tuning, faithfulness lifted from 0.62 to 0.74 — a 19% improvement on the same evaluation set with no changes to the generation model or prompt.

Eval-gated CI/CD — Blocking 20% of Regressive Updates

Regressive model updates were reaching production undetected because there was no systematic gate between a proposed update and deployment. I built an eval-gated CI/CD pipeline where every proposed model change must pass rubric-based LLM-as-judge scoring before it can proceed. The judge evaluates outputs against a curated test set using defined quality rubrics — scoring dimensions like groundedness, completeness, and format compliance. Updates that score below configured thresholds are automatically rejected with a detailed failure report. This pipeline blocked approximately 20% of proposed updates that would have caused quality regressions had they reached production.

001 Agentic · macOS
RAOC
Remote Autonomous OS Controller

RAOC lets you control your Mac through natural language commands sent from your phone via Telegram. You describe what you want done — rewrite a file, run a script, find a document and update it — and a pipeline of six specialised agents understands the request, locates the relevant files, builds a step-by-step plan, and presents it to you for approval before touching anything. Nothing executes without your explicit tap-to-approve.

  • 6-agent pipeline: Intake (understands intent), Discovery (finds and reads the target file), Planning (generates step-by-step action plan), Execution (carries it out), Verification (checks the result), Reporter (sends proof to your phone with screenshots and diffs).
  • Safety model: approval_granted enforced in code, not just convention. Dangerous command patterns (rm -rf, sudo, curl | bash, etc.) blocked by a pattern matcher before execution. Every file backed up automatically before any write. All file access sandboxed to a workspace folder — paths outside it raise an error at the zone resolver level, not just a warning.
  • Secrets handled properly: All API keys stored in macOS Keychain via keyring — never in code, environment variables, or .env files. Only your Telegram user ID is accepted; all other senders are silently ignored. Replay attacks mitigated by ignoring messages older than a configured threshold.
  • 303 passing tests covering every agent, the coordinator, command wrapper, zone resolver, and full database layer. GitHub Actions CI runs on every push.

002 Multi-agent
Bugfix AI Pilot
Autonomous Bug Resolution System

A multi-agent system that takes a JIRA ticket ID as input and — without any human involvement — reads the ticket, identifies the affected code, generates a fix, validates it by running the test suite, and opens a GitHub pull request with the changes and a summary. Each agent runs as an independent microservice communicating over REST APIs, coordinated by a central backend. A React frontend provides live monitoring and manual trigger capability.

  • Planner agent calls the JIRA API, reads the ticket description and acceptance criteria, uses GPT-4 to identify affected files and probable root cause, and outputs a structured repair plan with file paths and proposed changes.
  • Developer agent receives the plan, reads the relevant source files for context, and generates a code fix. Handles multi-file changes and maintains awareness of imports and dependencies.
  • QA agent runs the test suite against the generated fix. If tests fail, the failure output is fed back to the Developer for a second attempt. Up to three retry cycles before escalating to human review.
  • Communicator agent posts a comment to the JIRA ticket with the fix summary, opens a GitHub PR with the diff and test results, and links the PR back to the ticket.
  • Full-stack: React / TypeScript frontend with live ticket status cards, per-agent output logs, and a manual trigger form. All microservices in Docker Compose. 233 commits.

003 EvalOps
LLM Evaluation Harness
CI-ready A/B Evaluation Framework

A production-ready framework for A/B evaluating LLM changes — comparing two model versions, two prompts, or two configurations — with the statistical rigour and CI integration needed to make evaluation a real deployment gate rather than a manual spot-check. Built to reflect and generalise the EvalOps work I do at CGI.

  • Hard checks first: Deterministic validation runs before any LLM judge call — schema compliance, citation presence, output length, format requirements. Only cases that pass hard checks proceed to LLM evaluation, keeping costs down and judge attention on genuine quality questions.
  • LLM-as-judge with calibration: Rubric-based pairwise scoring. Judge calibration tools measure agreement rate, flip rate (consistency when A/B order is swapped), and order bias to verify the judge is reliable before trusting its outputs at scale.
  • Statistical rigour: Win rate reported with 95% bootstrap confidence intervals. Per-tag breakdown shows where one candidate wins and where it loses. Refusal rates and format pass rates tracked independently.
  • Quality gates for CI/CD: Configurable thresholds. If any gate fails, process exits with code 2, blocking the CI pipeline. Regression detection compares against a stored baseline and fails if win rate drops beyond a configurable tolerance.
  • Zero-cost CI: Mock providers run the full pipeline in GitHub Actions with no API keys, validating wiring and logic on every PR at no cost.
